# 我在这个项目里做了什么

先把话说在前面：这是两个人的团队项目，不是我一个人做的。我是全栈主力，整体架构是我定的，下面这些模块基本是我独立设计和实现的——AI 微服务（LangGraph agent、GraphRAG、异步记忆这一整套）、Java 核心后端（业务 + 管理端 RBAC + AI agent 的 Java 侧）、运维监控告警、数据管道，以及前端里 AI 对话和实时私信这两条核心链路。剩下一些页面和样式是和同学一起弄的。下面凡是说"我"的地方，都是我实际写的。

项目从 5 月底做到 6 月中，差不多一个月，挺密集的。最后是六个服务、五万行代码上下，后端 33 个 Controller、约 38 张表，AI 服务 18 个接口，小程序 32 个页面。

下面挑几件我自己最有体会的事讲讲，每件都尽量说清楚当时是怎么想的、为什么这么选。

## 让 AI 找帖别瞎答

这是我在这个项目上花时间最多的一块，也是我最满意的。

AI 学长有个功能是帮你找帖子，比如你问"有没有人卖二手自行车"，它去站内的帖子里给你捞相关的。一开始上线，它会把一条内容只有"测试"两个字的帖子也当成匹配返回——典型的幻觉，体验很差。

我一开始的反应跟很多人一样，想着是不是相似度阈值设低了，调高一点不就行了。试了之后发现不行：阈值调高，那些真正相关但描述得没那么"标准"的帖子也被一起误杀了。我去翻了实际的相似度分布才搞明白真正的原因——向量检索（我们用的 bge-m3 embedding）对中文短文本的相似度普遍偏高，一条内容是"测试"的帖子对"自行车"的余弦相似度也能到 0.7 左右，跟真正相关的帖子分数拉不开，光靠卡分数这条路是走不通的。

后来我换了个思路：与其在"分数"上较劲，不如再请大模型判一道。但提问方式很关键——原来的实现是"先假设这些帖子匹配，让模型给个匹配理由"，这种问法天然偏向于"找理由说它匹配"。我把问法反过来，变成"这几个候选里，哪些是真的能满足用户诉求的"，让它从严判断、宁缺毋滥，直接吐一个真正相关的帖子 ID 列表回来。这个判定不可用的时候（比如模型超时）才退回去用分数阈值兜底，两者是二选一不是叠加。再加上最后合成答案时也压一道"证据里没有的别编、找不到就老实说没找到"。

跑评测金标集，判定之后的 precision 到了 0.97，那些"绝对不该出现"的帖子漏出率是 0，知识问答的对抗样本拒答率是 100%。这套改动我都加进了评测基线做回归，后面再改提示词或检索参数，跑一遍就知道有没有把它改坏。

顺带说个我觉得挺有意思的小坑：评测里算 precision 的时候，我一开始是拿原始召回的结果算的，数字很难看。后来想明白了——向量检索本来就必然返回"最近的 topK"，它的模糊性不该算成"答错"，precision 应该算在"判定之后留下的帖子"这一层，因为那才是真正展示给用户的。这个口径是我自己跑评测的时候才反应过来的。

## 伺候内网那台单槽大模型

我们的对话模型（一个 35B 的 Qwen）跑在内网一台机器上，用的是 llama.cpp，配置是单槽——也就是同一时刻只能处理一个请求。这其实是整个 AI 链路真正的瓶颈，不是检索质量。压测的时候我发现整条链路（小程序 → Java → AI 服务）会排队雪崩，一个请求慢了后面全堵着。

我围着这个瓶颈做了三件事：

第一件，把 OpenAI SDK 的自动重试关掉。这个是我 debug 了挺久才发现的——SDK 默认 `max_retries=2`，一个请求因为槽位被占而读超时（60 秒）之后，它会自己再重试两次，单次 60 秒的等待被放大成最长 180 秒，而且这些重试还会把更多请求怼回那个本来就在排队的槽位，越堵越死。我把它改成不重试、超时就直接快速失败。

第二件，给整轮对话加了个"墙钟预算"。我在每个 LangGraph 节点的边界上判断一下时间到没到，如果整轮快超时了，就主动降级：跳过最慢的看图步骤、不再做第二轮规划、直接拿已经检索到的证据拼个答案出来，而不是再压一次大模型。这个预算我特意设得比 Java 那边的读超时还低一点，让 AI 服务先优雅降级，而不是干等到 Java 抛硬错。这里我没有用"硬中断"，因为同步线程池里没法把一个正在阻塞的大模型调用掐断，强行 kill 会留下脏连接；在节点边界上判断虽然掐不断正在跑的那一次，但能止住最大的放大源——也就是一连串串行的模型调用叠加。

第三件，能省的模型调用都省掉。比如"匹配理由"原来是召回阶段就给每个候选都算一遍，但大部分候选会被上面那道判定丢掉，白算了；我把它挪到判定之后，只给最终留下的那一两条帖子算。再加了个缓存，只缓存确定性的调用（temperature 为 0 的那些），命中了就直接省掉一整次模型调用。

## 管理后台的权限

运营后台需要很细的权限——审核、封禁、改配置、分配角色，还要能按学校隔离数据。我做的一个明确决定是：后台账号和 C 端学生账号**物理隔离**，不复用同一张 user 表，连 JWT 密钥都用两套，并且在 token 里打了 `type=admin` 的标记。这样 C 端的 token 根本没法拿来访问后台接口，反过来也一样。我觉得这比"共用一套密钥、靠一个角色字段区分"要稳得多，安全边界是从根上隔开的。

权限码用 `module:action` 的形式（像 `moderation:queue`、`user:ban`），方法上加个 `@RequirePermission` 注解，拦截器反射读出来校验，支持 AND/OR 组合。再叠加一个 Redis 黑名单，能把已登录的管理员强制踢下线。

跟它配套的还有审计：所有写操作（通过、驳回、封禁这些）我用一个 AOP 切面统一记下来，`try-catch-finally` 包着主逻辑，不管成功失败都异步落一条审计日志，连失败的错误摘要和耗时也记。审计这件事我特意做成异步而且失败只打个警告，绝不能因为记日志把主流程拖慢或者搞挂。

## 几个不那么显眼但我自己挺得意的小设计

运营看板要统计日活、周活、月活。我没用 SQL 的 `DISTINCT` 去重（用户量一大成本就线性涨），而是用 Redis 的 HyperLogLog——每次鉴权通过的时候把用户 ID `PFADD` 进当天的 key，算周活月活就把最近 7 天/30 天的 key 做并集去重。HLL 的特点是单个 key 就十几 KB、有大约 0.8% 的标准误差，对看板这种趋势性指标完全够用，是个拿精度换内存和性能的典型权衡。

还有线程池。AI 帖子入库（要调用看图模型，慢）和长期记忆抽取这两个异步任务，我特意用了不同的拒绝策略。入库用 `CallerRunsPolicy`，队列满了就回灌到调用线程，保证不丢；记忆抽取用 `DiscardPolicy`，队列满了直接丢——因为记忆这轮没抽到下轮还能再抽，而它绝对不能回灌到用户请求的线程上去，那会拖慢用户收到回复的速度。同样是异步，丢得起的和丢不起的，策略得分开。

## 数据进库前的脱敏

知识库的数据来自校园墙的帖子和微信群导出，里面全是学生隐私。我把数据管道的脱敏当成"地基"来做：默认是 dry-run，只把脱敏后的结果写成本地文件供人工抽检，必须显式加 `--push` 才会真的写进知识库。这么设计是想把"手抖跑错"的默认后果从"泄露"降级成"只是落了个本地文件"。脱敏覆盖了 11 类 PII，处理的时候有不少细节，比如微信 @ 提及里可能夹着手机号，正则得一直吃到特殊分隔符才行；花名册替换得长名字优先，不然"张三丰"会被"张三"先截断。这块在前端/运维那篇里有代码。

## 老实说几句

数字方面我尽量标清楚来源：评测的 0.97、0.93 是金标集跑出来的实测峰值，实际的门禁阈值我设得略低一点（留了点抖动空间）；像"真匹配相似度 0.94""8% 的提及消息被整体掩掉"这种是我观察和估计的经验值，不是代码里的常量，面试要是问到我会照实说。

项目是完整能跑、也用真实数据导过的，但用户量是校内试用级别。我做的优化都是冲着"内网单槽大模型"这个具体瓶颈去的，不是大规模分布式那一套，这点我不想往大了说。
