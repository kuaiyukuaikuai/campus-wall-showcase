# 系统架构

这篇讲整个系统是怎么搭起来的：分了哪几层、各个服务各管什么、数据怎么流动。涉及内网地址、主机名、密钥的地方都抽象掉了。

## 整体分层

```
   微信小程序 (uni-app)                运营后台 (Vue 3 + Element Plus)
        │ HTTPS / WSS                       │ HTTP（同源反代 /api /grafana）
   ┌────────────────────────────────────────────────────────────┐
   │            Nginx 同源反代：/ SPA · /api · /grafana            │
   └────────────────────────────────────────────────────────────┘
        ┌─────────────────────────┴─────────────────────────┐
   ┌───────────────────────────────┐    ┌──────────────────────────────┐
   │ Spring Boot 后端 (Java 17)     │HTTP│ AI 微服务 (FastAPI+LangGraph) │
   │ 用户·帖子·私信·AI agent·审核   │───►│ 知识问答 / 找帖 / AI 发帖      │
   │ 管理端 RBAC                    │    │ Neo4j 图谱 + 向量检索          │
   └───────────────────────────────┘    └──────────────────────────────┘
        │                                      ▲ /index · /ingest-post
        │                          ┌───────────┴──────────┐
        │                          │ 数据管道（清洗+脱敏入库）│
        ▼                          └──────────────────────┘
   ┌────────────────────────────────────────────────────────────┐
   │   MySQL 8 · Redis 7 · MinIO · Neo4j 5（知识图谱 + 向量索引）   │
   └────────────────────────────────────────────────────────────┘
                          │ 指标采集
   ┌────────────────────────────────────────────────────────────┐
   │ Prometheus → Grafana（看板） / Alertmanager → 转发企微·钉钉    │
   └────────────────────────────────────────────────────────────┘
```

整体是前后端分离加一个独立的 AI 服务。后端是 Spring Boot 单体，按领域分包（用户社交、社区帖子、AI、管理端、基础设施）；AI 能力我单独拆成了一个 Python 服务，两边走 HTTP 解耦。

为什么 AI 单独拆出来而后端不拆微服务：一方面 LangGraph、Neo4j 驱动、大模型这套生态在 Python 里最顺，另一方面 AI 服务要跑在带 GPU 的机器上，跟业务后端的部署位置天然不同，拆开各自伸缩更合理。后端本身两个人维护、业务耦合也高，单体内部分好包就够清楚了，没必要为了拆而拆。

## 六个服务各管什么

| 服务 | 技术栈 | 干什么 |
|------|--------|--------|
| 核心后端 | Java 17 / Spring Boot 3.4.2 | 用户、帖子、评论、私信、AI 学长 agent、内容审核、管理端 RBAC 加运营看板 |
| AI 微服务 | Python / FastAPI / LangGraph / Neo4j | 知识图谱索引、知识问答、帮你找帖、AI 代发帖、异步记忆 |
| 小程序前端 | uni-app 3 / Vue 3 / TS | C 端学生用的微信小程序 |
| 运营后台 | Vue 3 / Element Plus | 给运营和审核人员的管理界面 |
| 数据管道 | Python ETL | 从 MySQL / 微信导出抽知识，清洗脱敏后推给 AI 服务建图谱 |
| 运维编排 | Docker Compose | 把数据层、监控、告警、反代一份 compose 管起来 |

## 几条关键的数据流

**用户问 AI 学长。** 小程序带着 JWT 调后端的统一入口，Java 这边先读一下短期对话记忆、做指代消解（把"它"还原成上文说的东西），然后规划，再去调只读的工具——找帖走 AI 服务的 `/match-posts`，知识问答走 `/query`。拿回结果后过一道相关性判定（反幻觉那块），最后严格基于检索到的证据合成答案，再把这一轮异步沉淀成长期用户记忆。对话记忆是双层的：短期上下文存 `ai_chat_record`，长期的稳定事实异步抽取后存 `ai_user_memory`。

**知识入库。** 数据管道从 MySQL 的帖子评论、或者微信导出里抽出"问题—参考答案"，清洗、脱敏、假名化之后，POST 给 AI 服务的 `/index`，服务端再做切割、用 bge-m3 算 1024 维向量、抽取实体，写进 Neo4j。

**私信。** 小程序和后端之间是一条 WebSocket 长连接（带 30 秒心跳和断线重连），实时推送；历史消息、发送、撤回、已读这些走 HTTP。

**监控告警。** 各个服务通过 exporter 或 actuator 暴露指标，Prometheus 每 15 秒抓一次，命中告警规则后交给 Alertmanager 做分组和抑制，再 webhook 给一个我自己写的转发服务，转成企业微信/钉钉的 markdown 推到群里。

## 数据存在哪

四种存储各有分工：MySQL 存业务主数据（约 38 张表），Redis 做多级缓存和点赞收藏时间线这类计数、还有异步记忆的 Streams，MinIO 存图片附件，Neo4j 存知识图谱和原生向量索引。

Neo4j 的图模型分两块。知识这块是 `Document`-`Chunk`-`Entity` 的结构，`Chunk` 上挂着向量，实体之间还有 LLM 抽出来的关系三元组。帖子那块是 `Post` 连到 `Item`（带向量，用来语义找帖），再加意图、分类、标签的关系。两个向量索引都是余弦、1024 维。这里有个我觉得值得一提的取舍：给向量检索切的块要小（命中才精准），给图谱抽实体用的块要大（上下文才完整），所以这两套切分是解耦的，跟 Microsoft GraphRAG 的思路一致。

## 部署和环境隔离

后端、本地模型推理、AI 服务是直接跑在宿主机上的（后端用 systemd 托管），中间件和监控栈用 Docker Compose 编排，容器访问宿主机服务统一走 `host.docker.internal`。机器上中间件和 AI 服务集中在一台高内存的推理机，本地模型只绑 localhost、不对局域网暴露；前后端在各自的开发机上连内网中间件。

环境隔离是 dev / test / prod 三套，但我们只有一台机器跑中间件，所以不是复制三整套，而是按每种中间件最省的办法分租：MySQL 用多库、Redis 用不同逻辑库号、MinIO 用不同 bucket，唯独 Neo4j 因为社区版不支持多库，只能起三个独立实例。这是我在"省内存"和"别互相串数据"之间权衡出来的方案。

## 安全这块的几条底线

- 密钥一律走环境变量，`.env` 不进版本库。
- 数据进知识库前强制脱敏，管道默认 dry-run，人工核对后才推送；AI 服务入库前还会再过一道脱敏（连看图模型生成的图片描述也一起脱）。
- AI 答题严格基于检索到的证据，找不到就明说没有，不编。
- 管理端权限是物理隔离的独立账号体系，后端才是鉴权的权威，前端只做界面上的隐藏。
