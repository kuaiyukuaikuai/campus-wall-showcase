# Campus Wall 校园墙

这是我和一个同学两个人做的一个校园社交平台，断断续续做了一个月左右。我在里面是全栈主力，整体架构和大部分模块是我设计和实现的，后端和 AI 服务这两块基本是我一个人写的。

整理这个仓库，是想把项目里我自己觉得做得还算有点东西、也确实踩了不少坑的几个地方讲清楚，方便面试的时候能聊得具体一点。完整的业务代码在团队的私有仓库里，这里放的是架构说明、几个核心模块的设计思路，还有一些脱敏过的代码片段——真实的配置、密钥、内网地址都没放进来。

## 它是干什么的

简单说，是给大学生用的一个匿名社区加信息互助的微信小程序。发帖、评论、私信这些常规功能之外，最核心的是一个叫"AI 学长"的功能：你可以用大白话问它校园里的事（比如"四级多少分能报六级"），让它帮你在站内找帖子（"有人卖二手自行车吗"），甚至直接对它说一句话、让它帮你把帖子起草发出来。配套还有一个给运营/审核人员用的后台，以及一整套监控告警。

代码量大概是这个规模：

| 部分 | 技术 | 量级 |
|------|------|------|
| 核心后端 | Java 17 / Spring Boot 3.4 | 285 个文件、约 2.4 万行，33 个 Controller、75 个 Service、约 38 张表 |
| AI 微服务 | Python / FastAPI / LangGraph / Neo4j | 约 5800 行，18 个 HTTP 接口，配了 19 个测试文件 98 个用例 |
| 小程序前端 | uni-app 3 / Vue 3 / TS | 约 1.6 万行，32 个页面 |
| 运营后台 | Vue 3 / Element Plus | 约 2800 行，14 个权限页面 |
| 数据管道 | Python ETL | 约 660 行 |
| 运维监控 | Docker Compose / Prometheus / Grafana | 一份 compose 编排 17 个容器 |

六个服务用 Git Submodule 多仓聚合管起来，加起来五万行左右。

## 架构长这样

```
   微信小程序 (uni-app)              运营后台 (Vue 3 + Element Plus)
        │ HTTPS / WSS                     │ HTTP（同源反代 /api /grafana）
   ┌──────────────────────────────────────────────────────┐
   │            Nginx 同源反代：/ SPA · /api · /grafana      │
   └──────────────────────────────────────────────────────┘
        ┌───────────────────────┴───────────────────────┐
   ┌─────────────────────────────┐      ┌──────────────────────────────┐
   │ Spring Boot 后端 (Java)       │ HTTP │ AI 微服务 (FastAPI+LangGraph) │
   │ 用户·帖子·私信·审核·RBAC      │ ───► │ GraphRAG 问答/找帖·AI发帖·记忆 │
   └─────────────────────────────┘      └──────────────────────────────┘
        │                                       ▲ /index
        │                            ┌──────────┴──────────┐
        │                            │ 数据管道（清洗+脱敏） │
        ▼                            └─────────────────────┘
   ┌──────────────────────────────────────────────────────┐
   │  MySQL 8 · Redis 7 · MinIO · Neo4j 5（知识图谱+向量）   │
   └──────────────────────────────────────────────────────┘
                          │ 指标采集
   ┌──────────────────────────────────────────────────────┐
   │ Prometheus → Grafana / Alertmanager → 转发企微·钉钉机器人 │
   └──────────────────────────────────────────────────────┘
```

后端是 Spring Boot 单体，按领域分包；AI 能力单独拆成一个 Python 服务，两边走 HTTP 解耦，这样 AI 服务能单独部署在带 GPU 的机器上。整套基础设施是一份 Docker Compose 拉起来的，配了 Prometheus 抓指标、Grafana 看板，告警走 Alertmanager 再转到企业微信和钉钉群。

## 我觉得值得聊的几个点

不想堆功能清单，挑几个我自己花了最多心思、面试也最能展开讲的地方：

- **AI 找帖怎么做到不"瞎答"。** 一开始它会把内容只写了"测试"两个字的帖子也当成匹配返回，是个挺典型的幻觉问题。我后来定位到根因不是阈值调得不对，而是向量检索本身的特性，换了个思路加了一道判定，才真正治住。这部分在 [docs/ai-service.md](./docs/ai-service.md)。
- **怎么伺候内网那台单槽的大模型。** 我们的对话模型跑在内网一台机器上，同一时刻只能处理一个请求，特别容易被打爆。我围绕这个瓶颈做了一串工程上的取舍（关掉 SDK 自动重试、给整轮加超时预算、砍掉不必要的模型调用）。
- **管理后台的权限体系。** 后台的账号和 C 端学生账号是物理隔离的两套，连 JWT 密钥都分开，避免学生的 token 越权访问管理接口。这部分在 [docs/backend.md](./docs/backend.md)。
- **数据进知识库前的脱敏。** 数据源里有大量学生隐私（手机号、学号、微信号），管道默认是 dry-run 不写库的，必须人工抽检确认后才推送，脱敏放在最前面强制执行。

## 几句实话

这是两个人的团队项目，我是全栈主力，主导了架构和绝大部分模块，但不是我一个人全包的——具体哪些是我做的，[那篇贡献说明](./docs/portfolio.md)里写清楚了。文档里出现的数字，我尽量标了是哪来的：评测跑出来的实测值、还是我观察/估计的经验值。项目目前是完整能跑、也用真实数据验证过的状态，但用户量是校内试用级别，我做的性能优化是针对"内网单槽大模型"这个真实瓶颈，不是百万级流量那种场景，这点我不想含糊。

代码片段都是从真实源码里摘出来脱敏的，删掉了无关细节，保留能说明思路的部分。
